# S3 Storage Structure

S3 is used as a raw data archive — every API response and batch file is written here before (or alongside) being processed into RDS or DynamoDB. This provides a reprocessing safety net if schemas change or bugs are found in the ingestion pipeline.

## Bucket

**`mlbstats-raw-{account_id}`** (private, versioning enabled)

---

## Layout

```
mlbstats-raw-{account_id}/
│
├── api-responses/
│   │
│   ├── games/
│   │   └── {season_year}/
│   │       └── {YYYY-MM-DD}/
│   │           └── {game_id}.json          # Full MLB Stats API game feed response
│   │
│   ├── players/
│   │   └── {player_id}/
│   │       └── {YYYY-MM-DD}.json           # Player detail + season stats snapshot
│   │
│   ├── teams/
│   │   └── {team_id}/
│   │       └── {YYYY-MM-DD}.json           # Team roster + stats snapshot
│   │
│   ├── standings/
│   │   └── {season_year}/
│   │       └── {YYYY-MM-DD}.json           # Full league standings snapshot
│   │
│   └── schedule/
│       └── {season_year}/
│           └── {YYYY-MM-DD}.json           # Daily schedule response
│
└── statcast/
    └── {season_year}/
        └── {YYYY-MM-DD}/
            └── pitches.csv                 # Raw pybaseball/Statcast export for the day
```

---

## Conventions

- **One file per API call**: Each archived file represents a single raw API response, unmodified. No merging or transformation before archival.
- **Date in key = date the data was fetched**, not necessarily the game date. For game feeds, both the directory date and the `game_id` identify the game unambiguously.
- **Versioning enabled**: S3 object versioning is on so that if the same key is overwritten (e.g., a mid-game re-fetch), prior versions are recoverable.
- **Lifecycle policy**: Raw files older than 2 years transition to S3 Glacier Instant Retrieval to reduce cost. Files for the current and prior season remain in S3 Standard.

---

## Reprocessing Pattern

If a bug is found in ingestion logic, Lambda functions can be triggered to reprocess from S3:

1. List all keys under the relevant prefix (e.g., `api-responses/games/2026/`)
2. Read each raw JSON file
3. Re-run the transform and write to RDS / DynamoDB

This avoids needing to re-call the MLB Stats API for historical data.
